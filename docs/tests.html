Preuve de déterminisme dans le navigateur

Tests reproductibilité,
preuve de falsification

Ces tests sont volontairement simples. Ils démontrent une propriété clé du RPO : le déterminisme opposable.
Même entrée → même sortie, hash SHA-256 recalculable, altération détectable.
Aucun serveur, aucun service externe : tout s’exécute localement.

Ouvrir le bac à sable Consultez les spécifications RPO

Important : la réussite de ces tests ne “prouve” pas une vérité. Elle prouve que le processus est auditable, et que les transformations deviennent explicables.

Portée de test
Intégrité • Lisibilité • Vérifiabilité

Vérifications : déterminisme, stabilité de l’extraction (dates/lieux) et reproductibilité SHA-256.

Ce que ça démontre, côté dirigeant : vous n’êtes pas contesté pour ce que vous avez décidé, mais pour ce que vous ne pouvez plus reconstruire, et donc défendre.

Matrice de test
Test Ce que cela vérifie Statut Notes
Déterminisme Une même entrée produit un bundle identique et un hash identique. Ne pas courir Exécute l’extraction 2 fois, compare la structure canonique et le SHA-256.
SHA-256 Le hachage est reproductible pour une même chaîne canonique. Ne pas courir Utilise WebCrypto SHA-256 en local (pas de backend).
Ancres Données de base extraites de manière stable (dates, lieux) : pas de réécriture. Ne pas courir Extraction conservatrice (heuristique minimale) : stabilité plutôt que “brillance”.

À retenir : ces tests ne valident pas un “outil”. Ils valident une frontière de confiance : le système ne peut pas “changer d’avis” à votre insu.
Pourquoi cela compte (en une phrase)

On ne vous conteste pas vos décisions. On vous conteste ce que vous ne pouvez pas reconstituer — et donc pas défendre.

Vous voulez le pilote complet (CNRS × TruthX) : cohérence interprétative, couches narratives structurées, fusion avancée des signaux et rapport probant lisible par l’humain, conçu pour les professionnels.
