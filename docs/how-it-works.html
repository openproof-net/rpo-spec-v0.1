How RPO works, in plain, enforceable steps.

RPO v0.1 is a minimal proof bundle format: it turns a narrative into a deterministic artifact that can be audited later, without rewriting history. It does not decide truth. It makes integrity measurable and transformations explainable.

1) The pipeline (deterministic)

The engine follows a strict separation of concerns: it structures, seals, and validates. No hidden inference, no “smart” ranking, no interpretation layer.

1Input

A short narrative (1–3 paragraphs). Optionally: evidence references and a PDF path.

2Structure

The narrative is mapped into a minimal JSON bundle with canonical fields (issuer, subject, created_at, narrative block, evidence list).

3Integrity anchor

A SHA-256 public hash is computed over a deterministic concatenation of core fields. Any modification changes the hash. That’s the point.

4Validation

Independent implementations can recompute the hash and validate required fields. If it matches, integrity is confirmed. If not, tampering is detectable.

2) The anchor (what exactly is hashed?)

RPO v0.1 uses SHA-256 over a deterministic payload built from core fields. This guarantees that different validators reach the same result.
