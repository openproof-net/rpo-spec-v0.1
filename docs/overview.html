RPO — Rapport Probatoire Ouvert

What the RPO Standard Does in Practice

This page gives a conceptual overview of the RPO (Rapport Probatoire Ouvert) standard: what problem it solves, how it is structured, and how implementers and reviewers are meant to use it.

1. What is the RPO?

A dual-format standard for digital evidence

The RPO defines how a piece of digital evidence must be packaged so that:

  • machines can verify integrity and structure via a signed JSON bundle,
  • humans can read a coherent narrative via a PDF report,
  • both are anchored by a public hash that can be checked by any registry.

The goal is simple: make truth measurable, power traceable, and coherence verifiable across institutions, courts, and research projects.

2. Who is it for?

Intended audiences

The RPO standard is designed for:

  • Engineers building forensic or reporting engines,
  • Legal teams and auditors who must trust the narrative,
  • Registries and platforms providing long-term anchoring,
  • Researchers studying evidence structures or narrative bias.

Any system that wants to issue verifiable, sealed reports can implement RPO.

3. Core guarantees

Integrity • Readability • Verifiability

Every RPO-compliant bundle must satisfy three conditions:

  • Integrity — any modification to the evidence bundle or the narrative becomes detectable through hashing and signatures.
  • Readability — humans and institutions can interpret the same narrative, with timestamps and context preserved in the PDF.
  • Verifiability — a public, testable JSON structure allows third parties to check coherence and re-compute hashes.

These guarantees are enforced by the RPO schema, the format spec, and recommended signature practices.

4. High-level flow

From raw data to sealed report

At a high level, an RPO implementation follows four steps:

  1. Collect structured evidence and metadata.
  2. Build the RPO JSON object (rpo.json).
  3. Generate the narrative PDF and link it via narrative.pdf_hash.
  4. Anchor the bundle using a public hash (public_hash) and optional signatures.

The detailed flow and actor roles are described in /spec/rpo-architecture.md.

5. Files in this specification

How the spec is organised

The GitHub repository openproof-net/rpo-spec-v0.1 ships three main documents:

  • Architecture: /spec/rpo-architecture.md — actors (producer, registry, verifier) and data flow.
  • Format: /spec/rpo-format.md — field-by-field JSON and PDF mapping, including registry anchoring rules.
  • Schema: /spec/rpo-schema.json — machine-readable JSON schema for validation.

In addition, the /examples/example-minimal/ directory provides a minimal working bundle.

6. Versioning & roadmap

RPO standard evolution

The RPO specification is released incrementally. Current milestones:

  • v0.1 — Draft
    Initial JSON + PDF bundle specification, shipped with minimal example and tests.
  • v0.5 — Public Comment
    Feedback from CNRS / GREYC researchers, legal experts, and institutional partners.
  • v1.0 — Stable Release
    Finalized standard with registry integration and governance signatures.
